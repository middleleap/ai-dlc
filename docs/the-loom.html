The Loom

A way for a regulated entity to build with AI.

Set it up once, then weave any solution on it. Two harnesses — discovery finds the right problem, delivery ships it under control — run across a frame of always-on controls, weaving to the institution's own context. The output is live, audit-ready software, built the same way every time.

The harnesses

A double diamond — find the right problem, then deliver it.

Each diamond opens up to explore, then narrows to decide. The first is about the problem; the second about the solution. They meet at a single artifact — the agreed problem statement, the PRD — and the process is non-linear by design: evidence found during delivery can legitimately send you back a diamond.

A discovery is allowed to fail: stopping the wrong problem early is a win, not a waste. The waist between the diamonds is enforced — no feature enters delivery without a gate-green hand-off.

Diamond 1

Discovery — evidence in, problem out.

Before a line of code, the harness turns a challenge into evidence and an agreed problem: signals logged with stable ids, themes that trace to signals, one falsifiable problem statement, a data-governance verdict against the risk register, and a low-fidelity prototype a stakeholder actually reacted to. Every step is an artifact; every artifact is gated.

D1 FRAMING D2 EVIDENCE D3 SCOPE D4 NO-SOLUTIONING D5 SYNTHESIS D6 DATA-GOVERNANCE D7 BRAND D8 TANGIBILITY D9 VALIDATION

The prototype boundary

Brand-real, behaviour-hollow. A low-fi wireframe tests the framing — it is disposable, about the problem, and never binds delivery. Rendered by a zero-dependency engine against the brand profile, so it can't go off-brand even by accident.

The two seams

The data-risk register (D6) and the brand profile (D7) are mounted, not embedded. Swap the seams and the same machinery runs discovery for a different organisation — nothing hard-codes an entity, a regulator, or a platform.

The hand-off contract

Problem, success measures, out-of-scope, the inherited data-governance verdict, and the prototype as direction — with zero delivery design. Delivery inherits the risk position instead of rediscovering it.

Diamond 2

Delivery — an autonomous loop that proposes, never disposes.

One loop invocation takes one backlog item end to end. The agent authors and verifies; a human merges — every time. Compliance is a property of the line, not a scramble before an audit.

Pick

First eligible item. The waist gate rejects any feature without a gate-green discovery hand-off.

Isolate

One story, one branch, one session. Concurrent loops use isolated worktrees.

Spec-first

Contract path → generated types → contract tests, shown red before any implementation.

Implement

To green, against interfaces. Synthetic data only; audit and lineage in the same change.

Review

Reviewer agents on the diff: hard-stops PASS, contract CONFORMANT — or it doesn't proceed.

Human merge

PR opens, CI gates run, and the loop stops. Four-eyes is human, enforced by branch protection.

Deploy

Auto-deploy on merge; a smoke suite runs against the live URL and fails the deploy if broken.

Evidence

On release: gates re-run at the released commit; the evidence bundle is sealed into the repo.

Q1 BUILD Q1b TEST INTEGRITY Q2 STATIC + SAST Q3 INTEGRATION Q4 SECURITY Q4.5 LINEAGE Q5 PROD APPROVAL

Three tripwire hooks guard every edit: a PII guard blocks real-shaped identifiers before they're written, a spec tripwire locks the API contract mid-feature, and a test tripwire blocks weakened tests. The loop's discipline is enforced at the moment of the edit — it never depends on the agent's restraint.

The pattern

The context brain — why the method compounds.

A harness alone is generic, and so is an AI agent. What makes the Loom produce this institution's software is context, treated as a first-class governed asset — access-controlled, versioned, auditable, with owners and a change history. Not loose notes in people's heads.

Regulated context

What you must obey: mandates and deadlines, security posture, the risk taxonomy and controls. Lives as the data-risk register seam — discovery's D6 gate reads it on every run.

Solution domain

What you're building: the API contract, the data model, personas and scopes, the synthetic dataset. Swap this dimension and the same Loom builds a different solution.

Institutional DNA

How the entity works: approval routes, architectural patterns, approved technologies, design tokens, terminology, tone of voice. The part no competitor shares. Its institution-owned seed is the Institutional BrainKit — a versioned, approved, digest-pinned package every repository inherits before it writes a line.

The moat test. If a competitor copied the entire codebase tomorrow, what would they still lack? Everything that makes it yours — the accumulated, governed context. It cannot be copied, only built, cycle after cycle. And bad context compounds as readily as good: curation is a control, not housekeeping.

Always-on

Continuous assurance — agents, not meetings.

Keeping a regulated entity compliant traditionally means an endless calendar of review, risk, and sign-off meetings — assurance only as fresh as the last one. The harness pulls specialised agents through the same lifecycle continuously: on every change, on a schedule, and on events.

The loop never ends — it re-runs on the next trigger, and every cycle writes back to the brain. Humans are not removed; they are reserved for judgement and the approvals that must stay human.

Who assures the AI

Governance — AI proposes; humans and a protected control plane dispose.

An ungoverned agent can author, self-review, self-merge, deploy, and edit its own guardrails. The HG catalog is the set of decisions that closes each of those gaps — written as real ADRs by the adopting institution, each naming its enforcement of record.

HG-0001 Self-merge, no human four-eyes

Human-approved merges via enforced branch protection. The agent authors and verifies; a human disposes. The agent identity is not in the approver group and can't toggle the protection.

Loom 2.0 — mandate to customer outcome

One traceable control chain, from mandate to outcome.

Beyond building software right, Loom 2.0 adds a bank product-governance plane: a change's risk classification compiles the gates, control functions, and evidence it must carry, so a documentation fix and a lending-decision model take different routes by construction. The full gate set is executed by a risk-scoped runner that records every skip. Each stage below is shipped machinery with a negative bypass test.

Product assurance (PA1 · PA2)

A product cannot enter Develop without permission-to-develop, or reach launch without permission-to-launch — sections compiled per jurisdiction and product type, approvals resolving to authenticated humans holding the role. The Loom orchestrates and evidences the bank's product approval; it does not replace it.

Architecture assurance (A1–A5)

Data & privacy, a threat model that includes the AI harness itself with threat→control→test traceability, operational resilience, model risk, and core financial integrity — a material open finding blocks backlog creation.

Production authorization

Compound, not "a human approved": PA2 + freshness-windowed operational readiness + the second-line release hold released by a second-line human (missing = held) + externally-anchored, cryptographically-verified evidence at high risk.

Identity-bound evidence

Every approval and validation resolves against a registry — agents prepare evidence but approve nothing, and second-line membership is disjoint from the builders. A name in a text field does not count.

Runtime governance

High-risk models declare monitoring, suspension thresholds and a fallback; a signed continuous-assurance cycle record and an append-only agent decision log make the running system examinable, not just the build.

Vendor-neutral by seam

Enterprise adapters map branch protection, GRC registers, vaults and incident tools into the same signed-envelope currency, so the core runs on a different platform stack without changing a gate.

The proof — and the limits

Proven on Open Finance. Stated plainly.

The first build on the Loom was a bank-neutral UAE Open Finance back office: an autonomous loop carried 134 of ~139 backlog stories to done under the full gate set, every merge by human four-eyes, on synthetic data — and the discovery harness took real problems end to end through all nine gates. A method worth adopting also names its own risks:

Demo-proven, pilot pending

The proof runs on synthetic data. The bundle-side controls are complete through the 2.0 release candidate — the supervised-pilot playbook and its adversarial checklist ship — but the pilot itself, an independent risk review, and internal-audit re-performance are the adopter's to run. Adoption is not regulatory certification.

"Any domain" is early evidence

One bounded greenfield domain proved the loop. Legacy integration, real data, and organisational change are the true costs elsewhere — and are not yet in the cost curve.

The brain must be curated

Bad context compounds as readily as good. The brain is governed, versioned, and quality-checked — curation is a control, not housekeeping.

AI model risk is real

The agents need ongoing validation, monitoring, and independent challenge — that is what the catalog's model-risk decisions exist for.

Adopt it

The Loom ships as a plugin.

The method canon, the gate validator, the branded renderer, the artifact templates, the build-loop skills, the reviewer agents, and the guardrail hooks — packaged for Claude Code, adopted into a repository in one guided pass.

// in Claude Code
/plugin marketplace add middleleap/ai-dlc
/plugin install middleleap-loom@middleleap-ai-dlc

// then, in the repository that adopts the method
/middleleap-loom:loom-adopt

Installing adds two skills and two reviewer agents — nothing always-on. The hooks and the build loop activate only when a repository adopts them explicitly. The machinery is verified in CI by a full adoption dry-run: the bundled test suites must pass in the adopted layout on every push.
