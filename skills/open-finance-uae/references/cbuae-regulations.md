# CBUAE Open Finance Regulation

## Regulation Reference
- **Circular**: C 03/2025
- **Issued**: 10 July 2025
- **Status**: In-Force
- **Replaces**: Circular No. 7/2023 (31 December 2023)
- **Source**: https://rulebook.centralbank.ae/en/rulebook/open-finance-regulation-0

---

## Regulatory Objectives

The CBUAE's objectives under this Regulation:
1. Ensuring safety and soundness of Open Finance Services
2. Effective and risk-based licensing for Data Sharing and Service Initiation
3. Promoting reliability, efficiency and public confidence
4. Encouraging innovation to promote competition and benefit consumers
5. Reinforcing UAE's status as a leading fintech hub

---

## Mandated Entities

Participation is **mandatory** for all Licensees. The following are mandated:

| Entity Type | Requirement |
|-------------|-------------|
| Banks incorporated in UAE | Mandatory |
| Branches of foreign banks / representative offices | Mandatory |
| Specialized banks | Mandatory |
| Restricted licence banks | Mandatory |
| Islamic banks and Islamic windows | Mandatory |
| Finance companies | Mandatory |
| Payment service providers (Category 1/2/3/4) | Mandatory |
| Retail payment system providers | Mandatory |
| Stored value facility providers | Mandatory |
| Exchange houses | Mandatory |
| Loan-based crowdfunding companies | Mandatory |
| Insurance brokers | Mandatory |
| Insurance companies (national + foreign branches) | Mandatory |
| Any other entity deemed relevant by CBUAE | As determined |

**Phase 1**: Banks (including foreign branches) and Insurance Companies only.
Later phases to be announced by CBUAE through official channels.

## Products and Accounts in Scope (Article 5)

| Category | Products |
|----------|----------|
| Deposits | Deposits, savings accounts, term deposits |
| Payments | Payment accounts, credit/debit/charge cards, standing orders, direct debits |
| Stored Value | Stored value facilities, prepaid accounts, post-paid accounts |
| Credit | Credit, loans, personal finance, mortgages, secured loans |
| Foreign Exchange | FX accounts and products |
| Virtual | Virtual accounts for any of the above |
| Insurance | Life and general insurance products |

**Exclusion**: Products regulated by Securities and Commodities Authority (unless SCA approved).

## Open Finance Licence Requirements

### Minimum Capital (Article 6)
- **Standard**: AED 1,000,000
- **Additional**: CBUAE may impose higher based on risk/size/complexity
- Must maintain at all times (Article 7)
- Must be higher of: minimum capital OR estimated wind-down costs

### Capital Instruments (Article 8)
**Included**:
- Paid-up capital
- Reserves (excluding revaluation reserves)
- Retained earnings

**Deducted**:
- Accumulated losses
- Goodwill
- Other items as determined by CBUAE

**Important**: Capital cannot be allocated from other regulatory capital requirements.

### Professional Indemnity Insurance (Article 9)
| Coverage Type | Minimum Amount |
|---------------|----------------|
| Single claim | AED 5,000,000 |
| Aggregate (annual) | Higher of AED 5,000,000 or 50% of annual Open Finance revenue |

**Must cover**:
- Unauthorized transactions
- Data loss and breaches
- Cyber security risks
- Delayed or incorrectly initiated transactions

---

## Persons Deemed Licensed (Article 3)

No separate Open Finance Licence required for:
1. Banks licensed under Central Bank Law
2. Finance companies under Finance Companies Regulation
3. Retail Payment Service providers under RPS Regulation
4. Insurance brokers under Insurance Law
5. Insurance companies under Insurance Law
6. Stored value facility providers under SVF Regulation

**Requirements for Deemed License**:
- Prior written notice to CBUAE
- Description of intended services and resources
- Governance arrangements
- CBUAE approval before commencing (60 working days decision period)
- Silence = implicit rejection
- Full compliance with all Regulation articles once approved

---

## Limitations on Open Finance Providers (Article 4)

**Prohibited Activities**:
1. Receive, hold or transfer funds for users
2. Provide Advice on products/accounts
3. Make personalized recommendations
4. Receive fees/commissions from product providers
5. Process Sensitive Data (health, genetic, sexual, psychological)
6. Negotiate, mediate or effect transactions
7. Engage in insurance intermediation or underwriting

**Exception**: Providers with additional relevant licenses may perform restricted activities.

**Permitted**: Providing non-specific information/analyses on commercially available products (displayed online or otherwise, but not Advice).

## Data Sharing Obligations (Article 16)

### Data Holder Must:
- Communicate User Data as requested
- Treat requests same as direct user requests
- Communicate securely per Open Finance Framework

### Data Sharing Provider Must:
- Only share data per explicit user consent
- **Never process Sensitive Data** (even with consent)
- Ensure credentials not accessible to other parties (except issuer)
- Transmit credentials through secure channels
- Identify itself to Data Holder and User
- Not use/store data beyond explicit service requirements

---

## Service Initiation Obligations (Article 17)

### Service Owner Must:
- Communicate securely with Service Initiation Provider
- Immediately provide all information for transaction initiation
- Display transaction status until completion
- Treat instructions same as direct user instructions

### Service Initiation Provider Must:
- Only initiate per explicit user consent
- Ensure credentials protected and transmitted securely
- Identify itself to Service Owner for each transaction
- Not use/store data beyond explicit service requirements

---

## Authentication Requirements (Article 18)

- **Minimum**: Two-factor authentication (knowledge, possession, or inherence)
- **Higher risk**: Additional procedures required
- Data Holders/Service Owners select and implement procedures
- TPPs may rely on LFI authentication

---

## Consent Requirements (Article 22)

### Consent Must Be:
- Specific to purpose
- Informed and unambiguous
- Freely given
- Clear affirmative statement or action
- Separate consent for each purpose (if multiple)
- **Maximum 12 months** for recurring transactions
- Withdrawable at any time, as easy as giving consent

### Service Initiation Consent Must Include:
- Relevant Account(s)/Product(s)
- Nature of transaction (including if recurring)
- Transaction value(s)
- Beneficiary(ies)
- Value date(s)

### Invalid Consent:
- If includes Personal Data not relevant or necessary
- If includes third-party Personal Data without their consent or anonymization

---

## Data Privacy and Storage (Article 22)

### Data Processing Principles:
1. Lawfully, fairly, transparently
2. Collected for specified, explicit, legitimate purposes
3. Adequate, relevant, limited to necessary
4. Accurate and kept up-to-date
5. Appropriate security measures

### Data Residency:
- **Requirement**: All Open Finance data stored in UAE
- **No copies outside UAE** unless:
  - CBUAE approval + other competent authority approvals
  - Prior written user consent (informed of location, reason, legal risks)
  - Written user acknowledgment of foreign legal access risks

### Foreign Bank Branch Exception:
- May store outside UAE if:
  - CBUAE approval
  - Daily-updated Master System of Record copy maintained in UAE

### Data Destruction:
- Must destroy identifiable Personal Data after purpose completed
- Subject to 5-year record retention (Article 13)
- Subject to AML Laws retention requirements

---

## Liability Framework (Article 21)

### Open Finance Provider Liable For:
- Unauthorized access to User Data
- Loss of User Data held by provider

### Service Initiation Provider Liable For:
- Non-execution, late or defective execution
- Late initiation or processing
- Failure to ensure proper authorization/authentication
- Failure to accurately record transactions
- Failure to use appropriate secure communication
- **Burden of proof**: On provider to prove correct processing with evidence

### Service Owner Liable For:
- Non-execution, late or defective execution
- **Exception**: Not liable if caused by Service Initiation Provider

### Sanctions:
- Administrative and financial penalties for data breaches
- Without prejudice to other legal sanctions

---

## API Hub Requirements (Part II)

### Licensing (Article 23)
- Exclusive license from CBUAE required
- Cannot operate API Hub without license
- 3-year business plan required
- Must commence within 6 months of approval
- 6 months notice required for cessation/suspension
- CBUAE may suspend, withdraw or revoke license

### Minimum Capital (Article 25)
- **Requirement**: AED 20,000,000 paid-up capital (standalone basis)
- CBUAE may amend at sole discretion
- CBUAE approval for any capital changes
- Must immediately inform CBUAE if breach likely
- No objection required for profit/dividend distribution

### Governance (Article 26)
- **CBUAE representative on Board with veto right**
- Risk and Compliance Committee **chaired by CBUAE representative**
- CBUAE approval required for Board membership changes

**Required Board Committees**:
1. Strategy committee
2. Risk and compliance committee (CBUAE chair)
3. Audit committee (independent/non-executive, not merged with others)
4. Nomination and remuneration committee
5. Industry advisory committee

### API Hub Exclusive Activities (Article 23):
- Provide infrastructure/interface for Open Finance Framework
- Review and facilitate resolution of User disputes
- Issue operational and technical requirements including:
  - Digital access specification
  - Cyber security requirements
  - Customer journey design
  - Centralized consent management (including app-to-app)
  - **Right to implement capped charging or inhibit charging to TPPs**
- Activities authorized by CBUAE

### API Hub Licence Conditions (Article 24):
- Only conduct licensed activities
- Process only consented User Data necessary for services
- **Not permitted to access Personal Data or Confidential Data**
- Data protection controls with encryption
- Security incident handling procedures
- Business continuity management
- UAE data residency compliance
- Comply with Outsourcing Regulation (as if a "Bank")
- Comply with Principles for Financial Market Infrastructures

### API Hub Fees (Article 28)
- May charge fees for services
- Fees must be reasonable and justifiable
- CBUAE may impose caps or limits
- CBUAE may provide collection/transfer mechanisms

## Trust Framework Components (Schedule 1)

1. **Participants Directory**: 
   - Identity/access management for enrolled participants
   - Contact and enrollment management
   - Digital certificate validation
   - Application registration and validation

2. **Digital Certificates**: 
   - Minting, revocation, validation
   - Secure communication between participants

3. **API Portal**: 
   - Standards documentation
   - Technical specifications
   - Requirements and business rules

4. **Sandbox**: 
   - Ongoing testing
   - Official conformance certifications

---

## Common Infrastructural Services (Schedule 1)

1. **Consent and Authorization Manager**: 
   - Standalone app for users OR APIs for participants
   - Creation, management, enforcement, revocation of privacy directives

2. **Service Assurance**: 
   - Onboarding and registration request management
   - Technical enquiry handling
   - Data and transaction flow enablement

3. **Reporting and Analytics**: 
   - Operational data and KPIs across participants
   - Service performance, availability, adoption metrics

4. **Administration Tools**: 
   - Case and dispute management
   - Tracking, adjudication, resolution
   - Between participants or user-related

5. **Value Added Enablers**: As appropriate

## Corporate Governance (Article 11)

Required arrangements include:
- Board-approved organization structure
- Conflicts of interest controls
- Integrity and transparency controls
- Compliance controls
- Confidentiality and data privacy methods
- Regular monitoring and auditing procedures

**Authorized Individuals**: Must comply with Fitness and Propriety Regulation (Circular No. 4/2024).

---

## Risk Management (Article 12)

### Required Functions:
1. **Risk Management**: Independent, reports to Board, monitors operational/market/credit/legal risks
2. **Compliance**: Independent, reports to CEO with Board access, monitors laws/regulations/standards
3. **Internal Audit**: Independent, reports to Board, provides assurance on controls

### Required Framework Elements:
- Incident management procedures (detection, classification of major incidents)
- Business continuity and disaster recovery plans
- Business impact analysis
- Recovery strategies
- Alternative sites for business and IT recovery
- Software development lifecycle practices
- Sound administrative and accounting procedures
- Wind-down plan acceptable to CBUAE

### Outsourcing (Article 12.8):
- No material activity outsourcing without CBUAE non-objection (including related parties)
- Full responsibility retained by provider
- Core activities and key functions generally not permitted for outsourcing

---

## Record Keeping (Article 13)

**Must Maintain Records Of**:
- User consent for data access and transaction initiation
- All User Data received from Data Holders
- All transactions initiated
- Evidence of data destruction

**Requirements**:
- Kept securely in durable medium
- Available to CBUAE promptly on request
- **Retention**: Minimum 5 years from creation

---

## Notification Requirements (Article 14)

**Immediate Notification Required For**:
- Unjustified refusal of access by Data Holder/Service Owner
- Events preventing access or disrupting operations/security
- Legal action against provider or Senior Management (UAE or abroad)
- Insolvency, winding up, liquidation proceedings
- Appointment of receiver/administrator/liquidator
- Disciplinary measures or sanctions (by any body, UAE or abroad)
- Material regulatory changes beyond CBUAE
- Any violation or potential material violation
- Any other event specified by CBUAE

---

## LFI Obligations (Article 15)

Licensees who are Data Holders and Service Owners must:
1. Establish dedicated interface via API Hub for secure access
2. Register with Trust Framework within 14 days of CBUAE approval
3. Cooperate openly and timely with TPPs for Data Sharing/Service Initiation
4. **Not share User Data where user is not their customer**
5. **Not share User Data received from another Service Owner**

**Prohibited**:
- Data scraping or similar extraction activities
- Interception of digital connections via reverse engineering

---

## Secure Communication (Article 19)

All participants must use common and secure open standards for:
- Identification
- Authentication
- Notification
- Information exchange
- Security measures implementation

### Required Interface Capabilities:
1. TPPs can identify themselves to Licensees
2. Data Sharing Providers can securely request/receive Account/Product information
3. Service Initiation Providers can securely provide initiation and receive status

**Financial Messaging**: Must use ISO 20022 elements, components or approved message definitions.

---

## Obligations Towards Users (Article 20)

### Terms and Conditions Must:
- Be in clear, plain, understandable language
- Not be misleading
- Be available in Arabic and English (minimum)
- Provide 60 days notice for changes
- Allow termination at no charge if user rejects changes

### Must Include:
- Schedule of fees and charges
- Contact details (legal name, registered address, agent address)
- Communication channels
- Notification manner/timeline for unauthorized/delayed/incorrect transactions
- Liability information for unauthorized transactions
- Complaint procedures
- Dispute resolution information
- Unauthorized transaction reporting procedure

---

## AML/CFT Requirements (Article 30)

- Comprehensive AML/CFT policies, procedures and controls
- Compliance with AML Laws (Decree Federal Law No. 20/2018)
- Robust fraud control policies and systems
- Identification and access controls

---

## Technology and Cyber Security (Article 31)

Required:
- Technology and cyber security risk management framework
- IT governance framework (structure, controls, policies)
- Board responsibility for risk management framework
- API safeguards for data exchange security
- Adherence to API Hub security requirements
- Industry standards and best practices (as directed by CBUAE)
- Cybersecurity incident response and management plan
- Adequate skilled resources for threat identification and protection

---

## Enforcement and Sanctions (Article 34)

Violations may result in:
- Administrative sanctions
- Financial penalties
- As deemed appropriate by CBUAE

Applies to: Open Finance Providers, API Hub, and Licensees

---

## Consumer Protection (Article 35)

Subject to:
- Applicable consumer protection laws
- Implementing regulations
- CBUAE Regulations

---

## Controller Approval (Article 10)

- Prior CBUAE authorization required to become Controller (≥20% shares/votes)
- CBUAE considers likely influence on compliance
- Fit and proper requirements apply
- May impose conditions on disposal, acquisition, or voting

## Key Definitions (Article 1)

| Term | Definition |
|------|------------|
| Account | Account held by User with Licensee for in-scope Products |
| Advice | Opinion, evaluation, recommendation intended to influence product choice |
| API Hub | Centralized API Hub and Trust Framework licensed by CBUAE |
| Confidential Data | Data relating to identifiable User |
| Controller | Person with ≥20% shares or ≥20% vote control |
| Data Holder | Licensee holding User Data |
| Data Sharing | Online service providing consolidated User Data |
| Data Sharing Provider | Licensed juridical person for Data Sharing activities |
| Initiate | Electronic instruction to effect transaction OR communicate agreement |
| Master System of Record | All data for core activities, services, risk management, compliance |
| Open Finance Provider | Juridical person licensed for Open Finance Services |
| Person Deemed Licensed | Entity deemed licensed per Article 3 |
| Processing | Any operation on Personal Data (collection, storage, use, disclosure, etc.) |
| Sensitive Data | Health, physical, psychological, mental, genetic, sexual data |
| Service Initiation | Service of initiating transactions by electronic means |
| Service Initiation Provider | Licensed juridical person for Service Initiation activities |
| Service Owner | Licensee holding Account or Product for User |
| Technical Service Provider | Provides technical support (IT, networks, data processing) - no license required |
| Transaction | Act initiated by User to effect transfer, credit, debit, etc. |
| Trust Framework | Established per Article 2 and Schedule 1 |
| Unauthorized Transaction | Transaction without User consent |
| User | Person using Data Sharing or Service Initiation |
| User Data | Information relating to User for in-scope Accounts/Products |
